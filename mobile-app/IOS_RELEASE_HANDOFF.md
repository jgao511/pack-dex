## Scanner architecture and frozen assets

iOS deliberately uses the existing browser/WebKit implementation: DOM camera/photo input, frozen-A TFLite WASM CPU, browser preprocessing, local full-catalog index, and the existing OCR/fusion fallback. Android alone selects the existing native adapters. Platform selection uses Capacitor `getPlatform()` and `isNativePlatform()`; it does not use a user-agent check.

The native build copies these scanner resources from `public/scanner-ai` into the Capacitor bundle during `vite build --mode native` and `cap sync ios`:

- `frozen-a-62f2ff60.tflite`
- `catalog-embeddings-a851d797.f16`
- `catalog-embeddings.meta.json` (including card-ID order/model binding)
- `catalog-metadata.json`
- TensorFlow.js core, CPU, and TFLite browser runtime scripts
- scalar, SIMD, and threaded WASM runtime files and worker scripts

The asset resolver produces `capacitor://localhost/scanner-ai/...` URLs with a native `./` base. The build has no development-host dependency. Model and index fetch failures remain processing failures; they do not produce random candidates. Model C, training data, scanner fixtures, holdout photos, local caches, and Python environments are absent from the iOS scanner bundle.

## Privacy and security audit

`PrivacyInfo.xcprivacy` is an app-target resource. It declares no tracking and no app-level required-reason API. Based on code-visible behavior, it declares email address, Supabase user ID, and saved virtual pack/collection content as linked data used for app functionality. Capacitor core also supplies its own valid empty privacy manifest. Xcode must generate and inspect the aggregated privacy report before upload; Apple requires valid privacy manifests for submissions and uses them to describe collected data and required-reason APIs ([privacy manifests](https://developer.apple.com/documentation/bundleresources/privacy-manifest-files), [App privacy details](https://developer.apple.com/app-store/app-privacy-details/)).

Static network findings:

- Production API/CDN configuration is HTTPS.
- No `NSAllowsArbitraryLoads`, localhost Capacitor server, advertising framework, ATT description, or token logging was added.
- Scanner recognition reads the captured image locally; no scanner upload path is present.
- Frozen model/index/runtime assets are bundled locally.
- No service-worker registration is present in the mobile shell, so the installed app does not depend on a stale website service-worker cache.
- Supabase public client configuration remains supplied through Vite environment variables. No service-role key or private secret belongs in the app.
- The app requests Google Fonts at runtime and accesses Supabase, Cloudflare Turnstile, PackDex/CDN, TCGplayer, and YouTube destinations. The owner must include relevant third-party practices in the published privacy policy and App Store privacy answers.

Owner confirmation is still required for:

- Supabase retention, deletion, security/logging, IP-address, diagnostic, and subprocessors behavior.
- Cloudflare Turnstile and CDN operational logs, device/IP data, retention, and whether any data is linked.
- Whether any analytics, crash reporting, marketing, affiliate tracking, or tracking domains are enabled outside this repository.
- Whether TCGplayer links are affiliate links and whether click/conversion data is received.
- Final retention/deletion periods for accounts, collection data, support email, server logs, and backups.
- Whether PackDex targets children or will use a Kids Category/child-directed distribution.
- Confirmation that no scanner photo leaves the device in production traffic testing.

Do not add `NSUserTrackingUsageDescription` unless the owner confirms cross-company tracking that actually requires ATT.

## Authentication and links

The app offers guest mode and first-party email/password authentication only. There is no Google, Facebook, or other social login, so Sign in with Apple is not currently required under App Review Guideline 4.8 ([App Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)). Supabase sessions persist through its browser storage. Signup confirmation and password reset intentionally keep the existing HTTPS website callbacks; the published `/mobile-app/auth/callback` and `/mobile-app/reset-password` flow was not replaced with a custom scheme. No universal-link entitlement is needed for that current flow.

On native iOS, external HTTP(S) links are routed through the official Capacitor Browser (SFSafariViewController behavior); relative/internal navigation remains in the WebView and `mailto:` remains a system-handled link. The scanner component still stops camera tracks/preview on teardown, visibility loss, tips, and capture, and serializes restarts through its existing lifecycle guard.

Mac/device testing must verify signup, confirmation, login, logout, forgot-password/reset, guest mode, existing-session relaunch, Turnstile behavior under the Capacitor origin, keyboard/autofill, TCGplayer, YouTube, mail, privacy/terms, share links, and predictable back behavior.

## Appearance and safe areas

The existing mobile CSS already uses `env(safe-area-inset-top)` and `env(safe-area-inset-bottom)` for the shell, bottom navigation, scanner, modals, and fixed controls. No global Android/browser spacing was changed. The iOS launch screen is a dark, flat PackDex wordmark treatment without a gradient or stretched image.

The generated AppIcon remains a Capacitor placeholder. The repository has branded images only up to 512×512, including pre-rounded/translucent assets, and does not contain a suitable final 1024×1024 opaque square App Store icon. Provide a brand-approved 1024×1024 source with no transparency, pre-rounded corners, gradient, or yellow treatment before TestFlight. It was not silently upscaled.

Apple requires an accessible in-app policy and App Store privacy-policy URL ([App Review Guidelines](https://developer.apple.com/app-store/review/guidelines/)). The in-app modal is reachable and `https://www.pack-dex.com/privacy` returned HTTP 200 during this audit, but its copy does not yet cover retention/deletion sufficiently.
- Confirm the owner has rights/authorization for all Pokémon card artwork, names, logos, set data, and other third-party content, or obtain legal guidance and provide review documentation as appropriate. The fan-made/non-affiliation disclaimer is present but does not itself establish rights.
- Remove all visible “beta” or development/placeholder presentation from the App Store build and metadata when submitting; Apple expects complete production apps, functional links, and reviewable content ([review readiness](https://developer.apple.com/app-store/review/)).
- Complete App Store privacy questionnaire, age rating, export-compliance, content-rights, support URL, privacy URL, screenshots, description, and review notes using owner-confirmed facts.
- Provide a stable reviewer demo account populated enough to inspect account-only collection/value features, plus concise instructions for scanner, guest mode, simulated pack openings, and confirmation that there are no physical prizes or redeemable value. ## Mac/Xcode handoff checklist

1. Install a current App Store Connect-supported Xcode on macOS and clone `main` at the recorded commit.
2. Install the Node version supported by Capacitor 8, then run `npm ci` at the repository root and in `mobile-app`.
3. Add the production public Vite environment values locally; never commit Apple credentials, private keys, provisioning profiles, or Supabase service-role secrets.
4. From `mobile-app`, run `npm run build:ios` and `npm run validate:ios`.
5. Open `mobile-app/ios/App/App.xcodeproj` (or run `npm run cap:open:ios`) and allow Swift Package Manager to resolve dependencies.
6. Select the PackDex target, confirm bundle ID `com.packdex.app`, iOS 15.0, version 1.0, build 1, iPhone/portrait, permissions, privacy manifest membership, Release optimization, and automatic signing. Select the owner’s team locally.
7. Replace the placeholder AppIcon with the approved 1024×1024 source and validate the asset catalog.
8. Build Debug and Release for a simulator and a physical iPhone. Do not treat simulator camera behavior as physical-device proof.
9. Exercise camera and Choose Photo; inspect Web Inspector/network logs to prove model/index/WASM/workers load from `capacitor://localhost`, no scanner photo upload occurs, no localhost/CDN scanner dependency appears, missing artifacts fail safely, and memory remains acceptable.
10. Test scanner navigation/visibility lifecycle repeatedly and confirm exactly one restart after return.
11. Test guest, signup, confirmation, login, logout, reset, relaunch persistence, keyboard/autofill, privacy/terms, TCGplayer, YouTube, mail, and public share links.
12. Generate the Xcode privacy report, inspect every SDK manifest/signature, archive a Release build, validate the archive, and review generated entitlements/resources for duplicates or unexpected permissions.
13. Create the matching App ID/app record in the owner’s Apple account, increment the build number for every upload, complete TestFlight metadata/privacy/export questions, and upload from Xcode or Transporter. Apple documents Mac/Xcode/Transporter upload paths ([upload builds](https://developer.apple.com/help/app-store-connect/manage-builds/upload-builds/)).
14. Only after physical testing and owner approval, distribute to internal TestFlight testers; do not submit for App Review until the submission blockers above are closed.

Windows preparation does not prove Xcode compilation, simulator/physical-device behavior, signing, archive validity, privacy-report aggregation, TestFlight processing, or App Store acceptance.
